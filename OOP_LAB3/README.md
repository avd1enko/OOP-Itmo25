## Паттерны проектирования

### Strategy — расчёт стоимости заказа

- IPricingStrategy описывает общий интерфейс:

- StandardPricingStrategy — просто сумма всех позиций
- ExpressPricingStrategy — сумма + фиксированная доплата за экспресс
- DiscountPricingStrategy — оборачивает базовую стратегию и применяет скидку:

**Почему это Strategy:**  
Логику цены можно менять не трогая начальный класс Order: достаточно подставить другую реализацию IPricingStrategy

### State — состояния заказа

В конструкторе Order начальное состояние - Preparing, дальше состояния меняются при помощи соответствующих методов

**Почему это State:**  
Поведение при переходе по состояниям выделено в отдельные классы, логика вынесена из ORder

### Builder — пошаговая сборка заказа

Builder позволяет:

1. выбрать тип заказа (standard / express),
2. по шагам накидать в него блюда,
3. потом получить готовый Order

**Почему это Builder:**  
Создание заказа разбито на несколько шагов, в самом Order нет логики

### Abstract Factory — создание меню
StandardMenuFactory и KidsMenuFactory возвращает блюда разных меню

**Почему это Abstract Factory:**
Вместо ручного создания блюд, можно вызывать фабрику и получить список блюд

### Observer — реакция на смену состояния заказа
Заказ уведомляет всех обсерверов при смене состояния, но не знает, какая именно там реализация

## SOLID

### S — Single Responsibility Principle

Каждый класс отвечает за одну зону ответственности:

### O — Open/Closed Principle

Код открыт для расширения, закрыт для модификации:

- Можно добавить новую стратегию цены или состояние заказа, не меняя Order
- Можно добавить новый тип меню, реализовав IMenuFactory

### L — Liskov Substitution Principle

- StandardOrder и ExpressOrder полностью заменяют абстрактный order везде, где он 
- Любая реализация стратегии или состояния может подставляться вместо другой и бвть совместимой с с Order

### I — Interface Segregation Principle
Все интерфейсы малеьнкие и не содержат чего-то избыточного

### D — Dependency Inversion Principle
Order не зависит от конкретных реализаций, а только от интерфейсов, например со State, Order не зависит от конкретных состояний, а только от интерфейса, так же со стратегиями
